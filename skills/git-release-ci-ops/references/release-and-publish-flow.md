# Release And Publish Flow

Use this reference for changeset decisions, release readiness, tags, and npm publish expectations.

## Authoritative flow

1. Merge code to `main`.
2. Let the `main` pipeline verify and mirror `main` to GitHub.
3. Let GitLab open or update the `release/next` merge request.
4. Merge the release MR after reviewing generated version and changelog changes.
5. Let GitLab create package-scoped tags like `cli-vX.Y.Z` or `mcp-vX.Y.Z`.
6. Start the manual publish job from the matching tag pipeline.
7. Let GitLab mirror the published tag to GitHub.

## Changeset decision

Add a changeset when the change affects published output under:
- `packages/cli`
- `packages/mcp`
- `packages/core`
- `packages/shared`

Do not add a changeset for:
- docs-only updates
- skill or workflow maintenance
- CI-only changes
- mirror script fixes that do not change published package output

## Versioning rules

- Do not hand-edit package versions.
- Do not hand-edit release changelog entries generated by the release MR flow.
- Do not create manual tags as the normal path.
- Treat `chore(release): prepare npm release` as automation-owned.

## Standard verification set

```bash
pnpm run build
pnpm run release-status
pnpm run typecheck
pnpm run lint
pnpm run test
cd packages/cli && npm pack --dry-run
cd ../mcp && npm pack --dry-run
```

## Publish-specific reminders

- GitHub Actions are verify-only.
- GitLab is the publish control plane.
- Trusted publishing is configured per npm package.
- `NPM_TOKEN` is a fallback, not the preferred long-term model.
